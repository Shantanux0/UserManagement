Chapter 1
🌍 What is this project?
Understanding the big picture before diving into code
Think of this project as a digital phonebook for users. Instead of storing phone numbers, we store each user's name and date of birth (DOB). Anyone can ask our API: "Who is user #1?" and we'll respond with their name, birthday, and automatically calculated age.

What is an API?

🍕 Real-World Analogy

An API is like a waiter at a restaurant. You (the client) sit at your table. The kitchen (database) has all the food (data). You don't go into the kitchen yourself — you tell the waiter what you want. The waiter goes to the kitchen, gets your food, and brings it back. Our Go application IS the waiter.

What is REST?

REST (Representational State Transfer) is a set of rules for how APIs should behave. It uses standard HTTP methods:

GET

Fetch/Read data. Like asking "show me user #1"

POST

Create new data. Like saying "add a new user named Alice"

PUT

Update existing data. Like saying "change Alice's name to Alice Updated"

DELETE

Remove data. Like saying "remove user #1 from the database"

💡 Why use the "internal" folder?

In Go, code inside a folder named internal can ONLY be used by code within the same project. It's Go's way of saying "these are private implementation details — other projects cannot import them". This is a Go best practice for organizing large codebases.

💡 Why separate handler, service, and repository?

This is called Layered Architecture (also called N-tier or Clean Architecture). Each layer has ONE responsibility:
• Handler = "Receive the HTTP request, check if it's valid, send the response"
• Service = "Apply business rules (like calculating age)"
• Repository = "Talk to the database"

If you ever swap PostgreSQL for MongoDB, you only change the Repository. The Handler and Service stay the same!

Chapter 4
⚙️ sqlc.yaml — Database Code Generator Config
Tells the SQLC tool how to generate Go code from SQL queries
sqlc.yaml
🤖 Analogy

SQLC is like a robot assistant. You write SQL queries in plain SQL, and the robot reads them and automatically writes all the Go code needed to run those queries. You never have to write repetitive database boilerplate code by hand. sqlc.yaml is the instruction manual you give to the robot.

1
version: "2"
SQLC configuration version. SQLC has had multiple configuration formats. version: "2" tells SQLC to use its newer, more powerful configuration format (version 2 supports multiple SQL databases in one config).
2
sql:
SQL configuration block. Everything indented under this is defining ONE database target. If you had multiple databases (e.g., MySQL AND PostgreSQL), you'd have multiple list items here.
3
engine: "postgresql"
Database type declaration. Tells SQLC which database dialect to use. PostgreSQL has slightly different SQL syntax than MySQL. SQLC needs to know which one to target so it generates compatible code.
4
schema: "db/migrations"
Where the table definitions are. SQLC reads our .sql migration files here to understand what tables exist and what columns they have. Without this, SQLC wouldn't know that users has columns named id, name, and dob.
5
queries: "db/query"
Where the SQL queries are. SQLC reads the users.sql file from this folder. These are the SELECT, INSERT, UPDATE, DELETE statements that SQLC will convert into Go functions.
6-10
gen: go: ...
Code generation settings. Tells SQLC to generate Go code (as opposed to TypeScript, Python, etc.). The sub-fields configure how that Go code should look.
8
package: "db"
Go package name for generated code. All generated Go files will start with package db. This is what you type in import statements.
9
out: "db/sqlc"
Output directory. Generated Go files will be placed in the db/sqlc/ folder. When you run sqlc generate, these files appear here automatically.
10
sql_package: "pgx/v5"
Which PostgreSQL driver to use in generated code. SQLC can generate code for multiple drivers (lib/pq, pgx/v4, pgx/v5). We use the latest pgx/v5 because it's the fastest and most modern.
11-13
overrides: ...
Type mapping overrides. By default, SQLC might not know how to represent a PostgreSQL DATE column in Go. We tell it: "whenever you see a date type in the database, use Go's time.Time type". This is crucial for our DOB field.
Chapter 5
🗄️ Database Migrations — Creating the Table
SQL scripts that define and modify the database structure over time
db/migrations/000001_create_users_table.up.sql
🏗️ Analogy

A migration is like a construction blueprint change order. The .up.sql file says "build this structure". The .down.sql file says "tear it down if something goes wrong". The number prefix 000001 means this is migration #1 — later changes would be 000002, 000003, etc., keeping a history.

Up Migration (000001_create_users_table.up.sql)

1
CREATE TABLE IF NOT EXISTS users (
Create the users table. CREATE TABLE is SQL for "make a new table". IF NOT EXISTS is a safety guard — if you run this twice by accident, it won't crash. It'll just do nothing the second time. users is the table name.
2
id SERIAL PRIMARY KEY,
Auto-incrementing unique ID column. SERIAL = PostgreSQL automatically assigns the next number (1, 2, 3, ...) every time a new row is inserted. PRIMARY KEY = this column uniquely identifies each row. No two users can have the same ID.
3
name TEXT NOT NULL,
Name column. TEXT = stores any length of text (string). NOT NULL = this field is REQUIRED. If you try to insert a user without a name, PostgreSQL will reject it with an error.
4
dob DATE NOT NULL
Date of birth column. DATE is a special PostgreSQL type that stores calendar dates in YYYY-MM-DD format (e.g., 1990-05-10). Unlike TIMESTAMP which includes time, DATE is just the date. NOT NULL means it's required.
5
);
End of CREATE TABLE statement. The semicolon ; terminates the SQL statement.

Down Migration (000001_create_users_table.down.sql)

1
DROP TABLE IF EXISTS users;
Delete the users table entirely. DROP TABLE = completely removes the table AND all its data. IF EXISTS = don't crash if the table was already deleted. This is the "undo" for our up migration. Running this wipes all user data!
⚠️ IMPORTANT for Beginners

In real production systems, you NEVER delete data in down migrations — you only undo structural changes. Always be careful with DROP TABLE!

Chapter 6
🔍 SQL Queries — What We Ask the Database
The SQL commands that SQLC converts into Go functions
db/query/users.sql
These are the raw SQL queries we want to run against the database. SQLC reads these, combines them with the schema, and writes type-safe Go functions for each one. The special -- name: ... :one/:many/:exec comments are SQLC directives — they tell SQLC what to name the function and how many rows it returns.
1
-- name: CreateUser :one
SQLC directive comment. The double-dash -- makes it a SQL comment (ignored by the database). But SQLC reads it! name: CreateUser = the generated Go function will be called CreateUser(). :one = this query returns exactly ONE row.
2
INSERT INTO users (name, dob)
Insert a new row into the users table. INSERT INTO is SQL for "add a new record". We specify which columns we're providing values for: name and dob. We don't specify id because PostgreSQL generates it automatically.
3
VALUES ($1, $2)
Placeholder parameters. $1 and $2 are parameter placeholders. The actual values are passed in from Go code. $1 = first parameter (name), $2 = second parameter (dob). This pattern prevents SQL injection attacks!
4
RETURNING id, name, dob;
Return the newly created row. After inserting, immediately fetch back the complete row including the auto-generated id. Without RETURNING, you'd have to do a separate SELECT query to get the ID back.
6
-- name: GetUserByID :one
SQLC directive for fetching one user. The generated Go function will be GetUserByID() and it returns a single User struct or an error.
7-9
SELECT id, name, dob FROM users WHERE id = $1;
Find a specific user by ID. SELECT id, name, dob = get these specific columns (not * which would get all columns — being explicit is better practice). FROM users = from the users table. WHERE id = $1 = only return the row where the id matches our parameter.
11
-- name: UpdateUser :one
SQLC directive for updating a user. Returns one row (the updated user) after the update.
12-15
UPDATE users SET name=$1, dob=$2 WHERE id=$3 RETURNING...
Update existing user data. UPDATE users = modify a row. SET name=$1, dob=$2 = change these columns to the new values. WHERE id=$3 = ONLY update the row with this ID. Without WHERE, you'd update ALL rows! RETURNING gives us back the updated data.
17
-- name: DeleteUser :exec
SQLC directive for deletion. :exec means this query doesn't return any rows — it just executes a command. The generated Go function returns only an error (or nil if successful).
18-19
DELETE FROM users WHERE id = $1;
Delete a specific user. DELETE FROM users = remove a row. WHERE id = $1 = ONLY the row with this specific ID. Again, without WHERE, you'd delete EVERY user!
21
-- name: ListUsers :many
SQLC directive returning multiple rows. :many = this query can return 0, 1, or many rows. The generated Go function returns a slice ([]User) instead of a single User.
22-24
SELECT id, name, dob FROM users ORDER BY id ASC;
Get all users sorted by ID. No WHERE clause = get ALL rows. ORDER BY id ASC = sort results from smallest ID to largest (ASC = ascending). This ensures consistent ordering.
26-30
...LIMIT $1 OFFSET $2;
Paginated query. LIMIT $1 = return at most this many rows (e.g., 10). OFFSET $2 = skip this many rows from the beginning. For page 2 with limit 10: LIMIT=10, OFFSET=10. This is how pagination works in SQL!
Chapter 7
🤖 SQLC Generated Code — Auto-written Go
Code that SQLC writes for you — never edit these files manually!
db/sqlc/models.go & db/sqlc/db.go & db/sqlc/users.sql.go
✅ Key Insight

You never write these files yourself. Every time you run sqlc generate, SQLC regenerates them from scratch. If you manually edit them, your changes get overwritten next time. The first line of every generated file says: // Code generated by sqlc. DO NOT EDIT.

models.go — The Go struct matching our database table

1
// Code generated by sqlc. DO NOT EDIT.
Auto-generated warning comment. This literally tells you and every developer: don't touch this file manually. SQLC overwrites it every time you run sqlc generate.
5
package db
Package name. This matches what we set in sqlc.yaml: package: "db". All files in this folder belong to the db package. When you import them, you write import db "user-management/db/sqlc".
11-15
type User struct { ID int32; Name string; Dob time.Time }
The User struct — mirrors the database table. A struct in Go is a blueprint for a data object (like a class in other languages). SQLC created this to exactly match our users table: ID ↔ id SERIAL, Name ↔ name TEXT, Dob ↔ dob DATE (as time.Time because of our override in sqlc.yaml).

users.sql.go — Generated Go functions from SQL queries

13-17
const createUser = `INSERT INTO users...`
SQL stored as a Go constant. SQLC copies our SQL query into a Go constant string. The backtick (`) creates a "raw string literal" in Go, allowing multi-line strings and avoiding the need to escape quotes.
19-22
type CreateUserParams struct { Name string; Dob time.Time }
Parameter struct for CreateUser. SQLC created this struct to hold the $1 and $2 parameters for our INSERT query. Instead of passing loose variables, you pass a nice organized struct. Name maps to $1, Dob maps to $2.
24-29
func (q *Queries) CreateUser(ctx, arg) (User, error)
The generated CreateUser function. This is a METHOD on the Queries struct ((q *Queries) means it belongs to Queries). It takes a context and parameters, runs the SQL, scans the returned row into a User struct, and returns it. This is what you call from your repository code!
60-78
func (q *Queries) ListUsers(ctx) ([]User, error)
List function iterates through rows. Notice rows.Close() with defer (runs at end of function), and the for rows.Next() loop. This is the standard Go pattern for reading multiple rows from a database: iterate through each row, scan its columns into a struct, append to a slice, return the slice.
65
defer rows.Close()
Defer closes the rows cursor. defer in Go schedules a function call to happen when the surrounding function returns. rows.Close() releases the database connection. By deferring it, we guarantee cleanup happens even if an error occurs mid-loop.
Chapter 8
🚀 Migration Runner — Auto-runs Schema on Startup
The Go code that automatically applies pending database migrations when the server starts
db/migrations.go
💡 Why do we need this?

In production, you can't manually run SQL scripts on the database server. Instead, we embed the SQL files INTO the compiled Go binary and run them automatically when the app starts. This means the app is self-migrating — deploy it and the database is automatically up to date.

1
package db
Package declaration. This file belongs to the db package. Notice it's in the db/ folder, not db/sqlc/. This is handwritten code (unlike the sqlc folder which is auto-generated).
3-11
import ("context" "embed" "fmt" "strings" ...)
Imports. context = for cancellation/deadline propagation. embed = Go's built-in package for embedding files into the binary. fmt = formatted printing and error creation. strings = string operations like HasSuffix. pgxpool = database connection pool. zap = logger.
13
//go:embed migrations/*.sql
Go embed directive. This special comment (NOT a regular comment — Go treats //go:embed as a compiler instruction) tells Go to bundle ALL .sql files from the migrations/ folder INTO the compiled binary. The SQL files become part of the executable itself!
14
var migrationFS embed.FS
Embedded filesystem variable. embed.FS is a virtual filesystem. After compilation, migrationFS holds all the embedded SQL file contents in memory. You can read files from it using standard filesystem operations.
17
func RunMigrations(ctx context.Context, pool *pgxpool.Pool, logger *zap.Logger) error
Function signature. This function takes three inputs: ctx (context for cancellation), pool (database connection pool — the * means it's a pointer, not a copy), logger (for logging). It returns error (nil means success, non-nil means something went wrong).
20-29
CREATE TABLE IF NOT EXISTS schema_migrations...
Migration tracking table. Before running any migrations, we create a special table called schema_migrations that records WHICH migrations have been applied. This prevents running the same migration twice (which would cause errors). Each migration gets a version number stored here.
32
entries, err := migrationFS.ReadDir("migrations")
List files in the embedded directory. ReadDir returns a slice of directory entries (file listings) from our embedded filesystem. err captures any error. The err != nil check below is Go's idiomatic error handling pattern — always check for errors!
37-40
if entry.IsDir() || !strings.HasSuffix(entry.Name(), ".up.sql") { continue }
Skip non-migration files. We loop through all files. entry.IsDir() = skip subdirectories. !strings.HasSuffix(..., ".up.sql") = skip files that don't end in .up.sql (this skips .down.sql files). continue = skip to the next iteration.
43
_, err := fmt.Sscanf(entry.Name(), "%d_", &version)
Extract version number from filename. fmt.Sscanf parses a formatted string. The format "%d_" means "read an integer followed by underscore". From filename 000001_create_users_table.up.sql, it extracts 1 as the version number. The &version is a pointer — Sscanf writes the result INTO the variable.
49-53
pool.QueryRow(...).Scan(&exists)
Check if migration already ran. Queries the schema_migrations table to see if this version number already exists. EXISTS(SELECT 1 FROM ...) is a SQL trick that returns true if any matching rows exist, false otherwise. Scan(&exists) reads the result into our boolean variable.
67
tx, err := pool.Begin(ctx)
Start a database transaction. A transaction is a group of database operations that all succeed together or all fail together. If the migration SQL runs but recording the version number fails, the transaction rolls back BOTH changes — keeping the database consistent.
71
defer tx.Rollback(ctx)
Safety rollback. If anything goes wrong before tx.Commit() is called, this deferred Rollback() will undo any partial changes. After a successful Commit(), calling Rollback() is a no-op (harmless), so this is always safe to have.
77
tx.Exec(ctx, "INSERT INTO schema_migrations (version) VALUES ($1)", version)
Record that this migration was applied. After running the migration SQL, we insert the version number into schema_migrations. Next time the app starts, it'll see this version is already applied and skip it.
81
if err := tx.Commit(ctx); err != nil {
Commit the transaction. Commit finalizes all the changes made within the transaction, making them permanent. Only if Commit succeeds do we log "Successfully applied migration". If Commit fails, the deferred Rollback undoes everything.

⚙️ Configuration — Reading App Settings
How the application reads settings like the port and database URL
config/config.go
🎛️ Analogy

Config is like the settings panel of an app. Instead of hardcoding 3000 directly into your code (which means changing the code every time you deploy to a different environment), you put settings in environment variables. The same binary can run on your laptop (port 3000) AND in production (port 8080) without recompiling.

1
package config
Package config. This file is in the config package. Go packages are like namespaces — all code in this folder shares this package name, and other files import it as "user-management/config".
3-5
import ("os")
Import the os package. os is part of Go's standard library (built-in, no download needed). It provides access to operating system features, including reading environment variables with os.Getenv().
8-11
type Config struct { Port string; DatabaseURL string }
Config struct definition. A struct groups related data together. Our Config has two fields: Port (which port the server listens on, e.g., "3000") and DatabaseURL (the full connection string to PostgreSQL). Both are string type.
14
func Load() *Config {
Load function returns a Config pointer. func = function. Load() = function name (no parameters). *Config = the return type is a POINTER to a Config struct. The asterisk means "pointer to" — we return a reference to the struct, not a copy of it (more memory efficient).
15
port := os.Getenv("PORT")
Read the PORT environment variable. os.Getenv("PORT") reads the environment variable named PORT. If it exists (e.g., export PORT=8080), it returns "8080". If it doesn't exist, it returns an empty string "". The := is Go's short variable declaration (creates a new variable AND assigns its type automatically).
16-18
if port == "" { port = "3000" }
Default value fallback. If PORT wasn't set (empty string), default to "3000". This pattern — "try to read from environment, fall back to a default" — is standard practice for configurable applications.
26-29
return &Config{ Port: port, DatabaseURL: dbURL }
Create and return Config struct. The & before Config{} creates a pointer to the newly created struct. The Port: port syntax is Go's named field initialization — you explicitly say which field gets which value, making it very readable.
Chapter 10
📋 Logger — Structured Application Logging
Setting up Uber Zap for colorized, structured log output
internal/logger/logger.go
💡 Why not just use fmt.Println?

fmt.Println("Something happened") gives you plain text with no timestamp, no severity level, no machine-readable format. Uber Zap gives you: timestamps, log levels (DEBUG/INFO/WARN/ERROR), structured key-value pairs, extremely fast performance, and in development mode — beautiful colorized output!

3-6
import ("go.uber.org/zap" "go.uber.org/zap/zapcore")
Import Zap packages. go.uber.org/zap = the core logging library. go.uber.org/zap/zapcore = lower-level types and interfaces used for configuring encoders (how log messages are formatted).
9
var Log *zap.Logger
Package-level global logger variable. var declares a package-level variable (accessible from any file in the logger package). *zap.Logger is a pointer to a Zap Logger. Log is initially nil — it gets set when Init() is called.
13
config := zap.NewDevelopmentConfig()
Create a development logger configuration. Zap has two modes: Development (human-friendly, colorized, includes stack traces on errors) and Production (JSON format, minimal overhead, no stack traces). NewDevelopmentConfig() starts you with development defaults.
15
config.EncoderConfig.EncodeLevel = zapcore.CapitalColorLevelEncoder
Enable colorized log levels. The default encoder prints log levels like "info", "error" in plain text. CapitalColorLevelEncoder makes them INFO, WARN, ERROR in capital letters with terminal colors — much easier to scan!
18
Log, err = config.Build()
Build the logger instance. Build() applies all the configuration and creates the actual *zap.Logger instance. Note: we write Log, err = (not :=) because Log was already declared on line 9 — we're assigning to it, not creating a new variable.
19-21
if err != nil { panic(...) }
Panic on logger failure. panic() in Go immediately stops the program and prints a stack trace. We use it here (instead of returning an error) because if the logger fails to initialize, we can't even log the error — the app is fundamentally broken. Panicking early is the right call here.
Chapter 11
📐 Models — Data Shape Definitions
Go structs that define what JSON looks like going in and coming out
internal/models/user.go
💡 Why two different types?

We have request models (what the client sends us) and response models (what we send back). They're different because: (1) We don't want the client sending an id — that's auto-generated. (2) The GET response includes age which isn't stored — it's calculated. Having separate types for each makes this explicit and safe.

4-7
type CreateUserRequest struct { Name string `json:"name" validate:"required,min=1,max=100"` DOB string `json:"dob" validate:"required,datetime=2006-01-02,past_date"` }
Create User request struct with validation tags. Struct tags (backtick sections) are metadata attached to fields. json:"name" = when parsing JSON, the field labeled "name" maps to this Go field. validate:"required,min=1,max=100" = validation rules: field must exist, must be at least 1 character, at most 100.
6
DOB string `json:"dob" validate:"required,datetime=2006-01-02,past_date"`
DOB field with format and custom validation. datetime=2006-01-02 = the date MUST match this exact format. 2006-01-02 is Go's unique reference date for date formats (January 2nd, 2006 at 3:04:05 PM — Go uses this as a mnemonic). past_date is our CUSTOM validator that rejects future dates.
16-20
type UserResponse struct { ID int32; Name string; DOB string }

💡 Why have a Repository layer if SQLC already generates code?

SQLC generates code tightly coupled to PostgreSQL. The Repository layer adds an interface (a contract) on top. This means: (1) You can write unit tests by creating a "fake" repository that returns test data without hitting a real database. (2) If you switch to MySQL, you only change the repository — the service layer is unaffected.

Interface definition. In Go, an interface defines a set of method SIGNATURES (name + parameters + return types) without any implementation. Concrete implementation struct. This is the actual struct that IMPLEMENTS the UserRepository interface. It holds: queries (the SQLC Queries object for running SQL) and pool (the database connection pool). The * means these are pointers (no copying, just references).

Constructor function. In Go, there's no new keyword for classes. Instead, by convention you write a function named New... that creates and returns a configured struct. The CalculateAge Function — Most Important Logic

140
func CalculateAge(dob time.Time) int {
Function takes a time.Time, returns an int. This is a pure function — it has no side effects, doesn't touch the database, doesn't log anything. Given the same DOB, it always returns the correct age for today. Pure functions are the easiest to test.
141
now := time.Now()
Get the current time. time.Now() returns the current local date and time as a time.Time. We use this as the "as of" point for age calculation.
142
years := now.Year() - dob.Year()
Initial year difference. .Year() extracts just the year (e.g., 2026). Subtracting gives a raw year difference. For someone born in 1990, this gives 36. BUT this might be wrong by 1 if their birthday hasn't happened yet this year!
143
if now.Month() < dob.Month() || (now.Month() == dob.Month() && now.Day() < dob.Day()) {
The birthday-hasn't-happened-yet correction. Two cases where we subtract 1 year: (1) Current month is BEFORE birth month — their birthday is later this year. (2) Same month BUT current day is BEFORE birth day — their birthday is later this month. The && means AND, || means OR.
144
years--
Subtract one year. years-- is shorthand for years = years - 1. If the birthday hasn't occurred yet this year, reduce the calculated age by one year. Example: Born May 15, 1990. Today is January 12, 2026. Raw: 2026-1990=36. But Jan < May, so subtract 1 → age = 35. Correct!
146
return years
Return the calculated age. After the correction, years holds the correct age. Return it as an integer.

🚦 Analogy

Middleware is like a security checkpoint at an airport. Every passenger (HTTP request) must go through the checkpoint BEFORE reaching their gate (the handler). The checkpoint can add a stamp (request ID), measure how long you've been there, or even turn you away. After you're done and returning, you pass through again (logging how long the journey took).

Middleware factory function. This function RETURNS a handler (not IS a handler). The pattern func() fiber.Handler { return func(c *fiber.Ctx) error { ... } } lets you configure the middleware before registering it. It's a closure — the inner function has access to the outer function's variables.

Store request ID in request-scoped storage. c.Locals is like a temporary key-value store that lives for the duration of ONE request. We store the request ID here so the logging middleware (which runs later) can retrieve it without re-reading the header.

Record the start time. We capture the time BEFORE calling c.Next(). After the handler runs and control returns here, we'll subtract start from the current time to get the request duration.

Build structured log fields. zap.Field is a typed key-value pair for structured logging. zap.String("key", value), zap.Int("key", value), zap.Duration("key", value) create different field types. Using typed fields (vs string formatting) makes logs machine-parseable for log aggregation tools.

📦 Analogy

A Docker container is like a shipping container. The Dockerfile is the packing list — it describes exactly what goes into the container (OS, runtime, your code). Once packed, the container can run identically on your Mac, a Linux server, or in the cloud. No more "it works on my machine" problems!

💡 Multi-Stage Build

We use a two-stage build: Stage 1 (builder) uses the large golang image to compile the code. Stage 2 (final) uses the tiny alpine image and copies ONLY the compiled binary. The final image is tiny (~15MB instead of ~800MB) because it doesn't include Go compilers, source code, or build tools!

2
FROM golang:1.26-alpine AS builder
First stage: build environment. FROM specifies the base image. golang:1.26-alpine is an official Go image based on Alpine Linux (tiny, ~20MB). AS builder names this stage "builder" so we can reference it later. This stage has Go installed and can compile code.
4
WORKDIR /app
Set the working directory inside the container. All subsequent commands run inside /app. If it doesn't exist, Docker creates it. Like cd /app in the terminal.
7
COPY go.mod go.sum ./
Copy dependency files first (Docker cache optimization). We copy ONLY go.mod and go.sum before the source code. Docker caches each layer. If only source code changes (not dependencies), Docker reuses the cached downloaded dependencies layer — making rebuilds much faster!
8
RUN go mod download
Download all dependencies. RUN executes a command during the BUILD phase. go mod download downloads all packages listed in go.mod. This layer is cached — only re-runs if go.mod or go.sum changes.
11
COPY . .
Copy all source code into the container. Copies everything from the current directory (host) to the working directory (/app) inside the container. This runs AFTER downloading dependencies, so code changes don't invalidate the dependency cache.
14
RUN CGO_ENABLED=0 GOOS=linux go build -o main ./cmd/server
Compile the Go binary. CGO_ENABLED=0 = disable C bindings (creates a fully static binary with no external library dependencies). GOOS=linux = compile for Linux (even if building on Mac). -o main = name the output binary "main". ./cmd/server = compile this package. Result: a single executable file called main.
17
FROM alpine:3.19
Second stage: minimal production image. Starts fresh from the tiny Alpine Linux image (~5MB). No Go compiler, no source code, no build tools. Just the OS and whatever we explicitly copy into it.
22
COPY --from=builder /app/main .
Copy compiled binary from builder stage. --from=builder references the first stage by name. We copy ONLY the compiled binary (/app/main) into the current directory (. = /app). That's it — the entire application in one file!
25
EXPOSE 3000
Document which port the app uses. EXPOSE is documentation — it doesn't actually open the port. It tells Docker and developers "this container listens on port 3000". The actual port mapping happens in docker-compose.yml or with docker run -p 3000:3000.
28
CMD ["./main"]
Default command to run when container starts. CMD specifies the command that runs when the container starts. The array format (JSON format) is preferred over shell format — it runs the binary directly without a shell wrapper, which makes signal handling (Ctrl+C) work correctly.

🎼 Docker Compose — Multi-Container Orchestration
Running PostgreSQL and the API together with one command
docker-compose.yml
🎭 Analogy

Docker Compose is like a stage director for your application. The director says: "Database, you go first. Web app, wait until the database is ready, then start." It manages multiple Docker containers as a team, handling their startup order, networking, and configuration.

1
version: '3.8'
Docker Compose file format version. Specifies which Docker Compose feature set to use. Version 3.8 supports healthcheck, depends_on: condition, and other modern features we rely on.
3
services:
Services block. A "service" in Docker Compose is one container configuration. We have two services: db (PostgreSQL) and web (our Go API). Docker Compose starts both and connects them on a shared network automatically.
5
image: postgres:17-alpine
Use the official PostgreSQL 17 image. postgres:17-alpine is the official PostgreSQL image based on Alpine Linux (small). This is pulled from Docker Hub (Docker's public image registry). No need to install PostgreSQL manually — Docker handles it!
6
container_name: user_management_db
Give the container a friendly name. Instead of a random Docker-generated name, we can refer to this container by user_management_db. Also used as the hostname on the Docker internal network — so our API connects using hostname "db" (the service name).
8-10
environment: POSTGRES_USER: user_manager POSTGRES_PASSWORD: super_secret_password POSTGRES_DB: user_management
Configure PostgreSQL via environment variables. The official PostgreSQL image reads these special variables on first startup to: create a user named user_manager, set their password, and create a database named user_management. No SQL setup scripts needed!
12
- "5433:5432"
Port mapping: host:container. Maps host port 5433 to container port 5432 (PostgreSQL default). We use 5433 on the host to avoid colliding with a locally-running PostgreSQL on port 5432. Format is always host_port:container_port.
13-14
volumes: - postgres_data:/var/lib/postgresql/data
Persist database data. PostgreSQL stores its data in /var/lib/postgresql/data inside the container. Without a volume, all data would be LOST when the container stops. postgres_data: is a named Docker volume — it survives container restarts and recreations.
15-19
healthcheck: test: ["CMD-SHELL", "pg_isready ..."] interval: 5s timeout: 5s retries: 5
Health check configuration. Docker periodically runs the test command to check if the service is healthy. pg_isready checks if PostgreSQL is accepting connections. interval: 5s = check every 5 seconds. timeout: 5s = fail if no response in 5s. retries: 5 = mark unhealthy after 5 consecutive failures.
22
build: .
Build the web service from the Dockerfile. Instead of using a pre-built image, build: . tells Docker Compose to build an image using the Dockerfile in the current directory (.). Every time you run docker-compose up --build, it rebuilds this image.
28
DATABASE_URL: "postgres://user_manager:super_secret_password@db:5432/user_management?sslmode=disable"
Database connection URL. The format is: postgres://username:password@hostname:port/database?options. Note the hostname is db — which is the SERVICE NAME of the database container. Docker Compose automatically creates a DNS entry so db resolves to the database container's IP address.
29-31
depends_on: db: condition: service_healthy
Wait for database to be healthy before starting API. depends_on: db: condition: service_healthy tells Docker Compose: "Don't start the web container until the db container passes its health check". Without this, the API might try to connect to PostgreSQL before it's ready — causing startup failures.
33-34
volumes: postgres_data:
Declare the named volume. Named volumes must be declared here at the bottom. Docker manages the actual storage location. postgres_data: with no value uses Docker's default volume driver (local storage on the host machine).

🎓 How It All Fits Together
A complete walkthrough of what happens when you call POST /users

Complete Request Lifecycle — POST /users {"name": "Alice", "dob": "1990-05-10"}

STEP 1 HTTP request arrives at port 3000
STEP 2 Recover middleware wraps everything with panic protection
STEP 3 RequestID middleware generates UUID, sets X-Request-ID header, stores in locals
STEP 4 RequestLogger middleware records start time, calls Next()
STEP 5 Router matches POST /users → calls CreateUser handler
STEP 6 Handler: BodyParser reads JSON → {"name":"Alice","dob":"1990-05-10"} → CreateUserRequest struct
STEP 7 Handler: Validate.Struct() checks: name required ✓, min=1 ✓, max=100 ✓, dob format ✓, past_date ✓
STEP 8 Handler: calls service.CreateUser(ctx, req)
STEP 9 Service: time.Parse("2006-01-02", "1990-05-10") → time.Time object
STEP 10 Service: calls repo.CreateUser(ctx, "Alice", time.Time{1990-05-10})
STEP 11 Repository: calls SQLC queries.CreateUser() with CreateUserParams{Name:"Alice", Dob:...}
STEP 12 SQLC/pgx: Executes: INSERT INTO users (name, dob) VALUES ('Alice', '1990-05-10') RETURNING id, name, dob
STEP 13 PostgreSQL: Inserts row, auto-generates id=1, returns {id:1, name:"Alice", dob:1990-05-10}
STEP 14 Service: converts db.User to models.UserResponse, formats Dob as "1990-05-10" string
STEP 15 Handler: logs "Created user successfully id=1", returns c.Status(201).JSON(user)
STEP 16 RequestLogger middleware (after Next() returns): calculates duration, logs method/path/status/duration/requestId
RESPONSE HTTP 201 Created, X-Request-ID: [uuid], Body: {"id":1,"name":"Alice","dob":"1990-05-10"}
